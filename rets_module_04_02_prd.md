## 1. 제품 개요

### 1.1 모듈명

**RETS-MODULE-04: 도메인 모델링 및 용어 사전 구축 도구**

### 1.2 한 줄 설명

요구사항 문서로부터 핵심 도메인 엔터티·관계·용어를 LLM으로 자동 추출하고, 이해관계자 간 공통 이해를 지원하는 도메인 모델 및 용어 사전 구축 도구

### 1.3 배경 및 문제 정의

소프트웨어 개발 초기 단계에서 이해관계자 간 용어 혼선과 도메인 개념의 불일치는 요구사항 해석 오류의 주요 원인이다. 현재 많은 프로젝트에서 다음과 같은 문제가 반복적으로 발생하고 있다.

- 동일한 개념에 대해 팀마다, 문서마다 다른 용어를 사용함
- 도메인 모델이 체계화되지 않아 요구사항 분석의 일관성이 낮음
- 용어 정의 작업이 개인 역량에 의존하여 표준화되지 않음
- 완성된 도메인 지식이 문서화되지 않아 재사용이 불가능함

RETS-MODULE-04는 이러한 문제를 해소하기 위해, 요구사항 문서를 입력으로 받아 LLM 기반의 자동화된 분석을 통해 도메인 모델과 용어 사전을 구축하고 관리할 수 있는 내부 도구를 제공한다.

## 3. 제품(모듈) 범위

### 3.1 포함 범위 (In Scope)

- 개발 요구사항(DEV REQ) JSON 파일 로드 및 파싱
- LLM 기반 도메인 엔터티 자동 추출 및 사용자 검토·확정
- LLM 기반 엔터티 간 관계 자동 추천 및 사용자 확정
- Mermaid ER 다이어그램 기반 도메인 모델 시각화
- 사용자 직접 엔터티·관계 추가·수정·삭제 (CRUD)
- LLM 기반 용어 정의 자동 생성 및 용어 사전 등록
- 중복·충돌 용어 탐지 및 표준 용어 통합
- 용어 사전 검색 및 상세 조회
- 용어 사전 기반 요구사항 내 비표준 용어 탐지 및 수정 제안
- LLM 기반 도메인 모델 완성도 검토 및 개선 제안
- 용어 사전(Markdown/CSV) 및 도메인 모델 다이어그램(HTML/Markdown) 내보내기
- 한글/영어 다국어 지원

### 3.2 제외 범위 (Out of Scope)

- 백엔드 서버 및 데이터베이스 구축 (PoC는 프론트엔드 전용)
- RFP 요구사항(RFP REQ) 직접 입력 지원 (입력은 DEV REQ JSON 한정)
- 타 모듈(MODULE-01~03, 05)과의 실시간 API 연동
- 사용자 계정 관리 및 권한 제어
- 모바일 전용 최적화 UI

## 4. 핵심 유저 스토리 (RFP 기반 상세화)

### US-01. 요구사항 문서 불러오기 및 도메인 분석 초기화

**As** 요구사항 분석가,  
**I want to** DEV REQ JSON 파일을 도구에 불러와 도메인 모델링 작업 환경을 초기화하고 싶다,  
**So that** 요구사항 텍스트를 기반으로 도메인 분석을 시작할 수 있다.

**수락 기준:**
- RETS 공통 벤치마크 스키마(`rets_02c_req_schema.json`)를 준수하는 DEV REQ JSON 파일을 로드할 수 있다
- 로드된 요구사항 목록이 화면에 표시된다
- 스키마 오류 발생 시 오류 항목을 Toast로 안내한다
- 로딩 중 비동기 처리 UI(스피너 등)가 표시된다

---

### US-02. LLM 기반 핵심 도메인 엔터티 자동 추출 및 검토

**As** 요구사항 분석가,  
**I want to** 로드된 요구사항 텍스트를 LLM으로 분석하여 도메인 엔터티 후보를 자동으로 추출하고 싶다,  
**So that** 수동 분석 없이 빠르게 도메인 개념의 초안을 확보하고 검토할 수 있다.

**수락 기준:**
- [엔터티 추출] 버튼 클릭 시 LLM 호출이 비동기로 수행된다
- LLM은 엔터티 후보를 JSON 형식으로 반환한다
- 추출된 엔터티 목록이 사용자 검토 UI(수락/거부)와 함께 표시된다
- LLM 제안 외에 사용자가 직접 엔터티를 추가할 수 있다
- 확정된 엔터티만 도메인 모델에 반영된다

---

### US-03. LLM 기반 엔터티 간 관계 자동 추천 및 확정

**As** 요구사항 분석가,  
**I want to** 확정된 엔터티들을 기반으로 LLM이 관계(Relationship)를 추천해 주길 원한다,  
**So that** 도메인 모델의 관계를 빠르게 정의하고 시각화할 수 있다.

**수락 기준:**
- 엔터티 확정 후 [관계 추천] 버튼 클릭 시 LLM 호출이 수행된다
- LLM은 관계 목록을 JSON 형식으로 반환한다
- 추천된 관계에 대해 사용자가 수락/거부/수정할 수 있다
- 확정된 관계는 Mermaid ER 다이어그램에 즉시 반영된다
- 사용자가 직접 관계를 추가·수정·삭제할 수 있다

---

### US-04. LLM 기반 용어 정의 자동 생성 및 용어 사전 등록

**As** 요구사항 분석가,  
**I want to** 도메인 모델에 등록된 엔터티와 주요 개념에 대해 LLM이 비즈니스 맥락에 맞는 용어 정의를 자동 생성해 주길 원한다,  
**So that** 용어 사전을 빠르게 구축하고 이해관계자 간 공통 언어를 확립할 수 있다.

**수락 기준:**
- [용어 정의 생성] 버튼 클릭 시 LLM이 각 엔터티에 대한 용어 정의를 반환한다
- 생성된 정의는 사용자 검토 후 용어 사전에 등록된다
- 등록된 용어에 대한 CRUD(추가·수정·삭제)가 가능하다
- 동일하거나 유사한 의미를 가진 중복·충돌 용어가 자동 탐지된다
- 탐지된 충돌 용어는 이해관계자 합의를 통해 단일 표준 용어로 통합할 수 있다

---

### US-05. 용어 사전 검색 및 상세 조회

**As** 이해관계자,  
**I want to** 용어 사전에서 특정 용어를 검색하고 상세 정의와 관련 엔터티를 확인하고 싶다,  
**So that** 프로젝트 진행 중 언제든지 용어의 공식 정의를 참조할 수 있다.

**수락 기준:**
- 검색창에 키워드 입력 시 실시간으로 용어 목록이 필터링된다
- 용어 클릭 시 공식 정의, 관련 엔터티, 동의어, 비고 등 상세 정보가 표시된다
- 용어와 연결된 도메인 모델 엔터티로 이동하는 링크가 제공된다

---

### US-06. 용어 사전 기반 요구사항 내 비표준 용어 탐지 및 수정 제안

**As** 요구사항 분석가,  
**I want to** 용어 사전에 등록된 표준 용어와 다른 표현이 요구사항에 사용된 경우 이를 탐지하고 수정 제안을 받고 싶다,  
**So that** 요구사항 전체에서 용어 일관성을 유지할 수 있다.

**수락 기준:**
- 요구사항 목록에서 비표준 용어가 사용된 항목을 하이라이팅하여 표시한다
- 각 비표준 용어에 대해 용어 사전 기반 표준 용어 대체 제안이 제공된다
- 사용자가 제안을 수락하면 해당 요구사항의 텍스트가 표준 용어로 교체된다

---

### US-07. LLM 기반 도메인 모델 완성도 검토 및 개선 제안

**As** 아키텍트,  
**I want to** 현재까지 완성된 도메인 모델과 용어 사전의 품질을 LLM이 진단하고 개선 제안을 제공해 주길 원한다,  
**So that** 누락된 엔터티·관계·용어 정의를 파악하고 모델을 보완할 수 있다.

**수락 기준:**
- [완성도 검토] 버튼 클릭 시 LLM이 도메인 모델과 용어 사전 전체를 분석한다
- 누락된 엔터티, 부족한 관계 정의, 미등록 용어 등에 대한 구체적인 개선 제안이 표시된다
- 사용자가 개선 제안을 선택적으로 반영할 수 있다

---

### US-08. 도메인 모델 및 용어 사전 내보내기

**As** PM,  
**I want to** 완성된 도메인 모델과 용어 사전을 파일로 내보내고 싶다,  
**So that** 결과물을 다른 팀원 및 이해관계자와 공유하고, 후속 문서 작성에 활용할 수 있다.

**수락 기준:**
- 용어 사전을 Markdown 또는 CSV 형식으로 내보낼 수 있다
- 도메인 모델 다이어그램을 HTML 또는 Markdown 형식으로 내보낼 수 있다
- 내보내기 버튼은 Header 우측 영역에 표시된다

## 6. 사용자 경험 및 디자인 요구사항

### 6.1 화면 구조

모듈 UI는 RFP UI 요구사항(`rets_02b_design_requirements.md`)에 따라 다음 구조로 구성된다.

```
Header: [RETS 뱃지 + 모듈명] [조직 로고] [테마전환 | 불러오기 | 내보내기]
Working Tabs: [도메인 모델] [용어 사전] [요구사항 검토]
Body (2-column):
  Left Panel: 엔터티/관계 목록 + 편집 컨트롤
  Right Panel: Mermaid ER 다이어그램 시각화 / 용어 상세 / LLM 결과
```

### 6.2 핵심 UX 원칙

- LLM 응답은 자동 반영 금지 — 수락(Accept)/거부(Reject) UI 필수 제공
- 비동기 작업(LLM 호출, 파일 로드) 중 로딩 UI(스피너) 표시
- Alert 팝업 사용 금지 — 상태 알림은 우측 하단 Toast로 처리
- 되돌릴 수 없는 작업(삭제, 대량 반영)은 사용자 확인 모달 제공
- 진입 화면: 도메인 모델 현황 대시보드 (엔터티 수, 관계 수, 용어 사전 등록 수 요약)

## 7. 기술 고려사항

### 7.1 구현 제약

- **단일 파일**: Vanilla JS 기반 단일 HTML 파일 (`rets_module_04_07_app.html`)로 구현
- **백엔드 없음**: 서버/DB 없이 프론트엔드 기능만으로 동작 (로컬 스토리지 활용 가능)
- **iframe 통합 대비**: 추후 RETS 통합 앱에서 iframe으로 삽입되므로, 자체 완결 구조로 설계

### 7.2 LLM 연동

- LLM 릴레이 서버: `https://ai-relay.mooo.com:11443` (Google Gemini 기반)
- 참조 가이드: `/00_rfp_requirements/rets_02a_llm_relay_server_dev_guide.md`
- 호출 흐름: 세션 초기화(with 시스템 프롬프트) → 채팅 방식으로 요청/응답
- 시스템 프롬프트: `/00_rfp_requirements/rets_02a_llm_system_prompt.md` 참조
- LLM 요청 시 구조화된 JSON 반환을 명시적으로 요청하여 파싱 가능하도록 처리
- `team_id`는 UI를 통해 사용자가 입력할 수 있어야 함

### 7.3 도메인 모델 시각화

- Mermaid ER 다이어그램 라이브러리 활용 (`https://mermaid.ai/open-source/syntax/entityRelationshipDiagram.html`)
- 렌더링 실패 시 텍스트(Markdown) 형식으로 fallback 처리

### 7.4 데이터 저장

- 세션 내 작업 데이터는 JavaScript 객체로 메모리 관리
- 지속성이 필요한 데이터(사용자 설정 등)는 `localStorage` 활용 가능
- 최종 결과물은 파일 내보내기(Blob/download) 방식으로 저장
